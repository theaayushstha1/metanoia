These are talking points, not a paragraph to read word for word. Bold blue is the line to actually say. The rest is what to point at. The handwritten lines are the ones that land, so say them like you mean them.

1What it is0:00 · 30s
  • Metanoia is a storefront for agent‑led subscription commerce, built on Juspay Hyperswitch.
  • "You give an AI agent a spending mandate instead of your credit card. It shops, compares, and buys a subscription for you, and Hyperswitch settles the real payment in the sandbox."
  • The whole design in one line: the model proposes, the server decides, Hyperswitch settles. An LLM can recommend. It can never set a price or move money.

4The architecture doc (the PDF)4:30 · 1 min
  • "I framed it around the industry and the payment choices, in three pages." Walk the three figures.
  • Fig 1, the end‑to‑end flow. "The green boxes are the deterministic authority. The red dashed ones are deferred or refused, where the card is never touched."
  • Fig 2, the authority boundary. "Model proposes, server decides, Hyperswitch settles. Only a proposal crosses from the model to the server. Only the server can move money. The LLM has no payment tool."
  • Fig 3, the payment sequence. "SpendGuard runs again on the server before Hyperswitch is ever called. A refusal means a 403 and no payment intent."
  • Mention the trust boundaries: untrusted input wrapped as "data, not instructions," an SSRF‑guarded GitHub fetch, and the secret key kept server‑side only.
5The decisions doc (ADRs)5:30 · 1 min
  • "Every real decision is written down, with the alternative I rejected." The payment ones I would highlight:
  • Model proposes, server decides. The LLM only emits a structured proposal. decide() re‑validates on server‑owned prices and can override it.
  • SpendGuard before any charge. It checks the caps in order and returns 403 before Hyperswitch is contacted.
  • Fauxpay for checkout, Stripe reserved for recurring. "Fauxpay reliably settles the checkout. Real off‑session recurring needs a reusable payment‑method id, which only a real connector returns."
  • Stable payment IDs give idempotency. pay_ = sha256(customer:plan:period), so a retry reuses the intent and never charges twice.
  • Pending‑first renewal for crash durability, and atomic, HMAC‑verified webhooks that retain unknown events.
  • "Connectors are processors, not payment methods. The customer picks a card, and routing is my decision. Unified Checkout keeps me out of PCI scope."
6Webhooks and the honest limits6:30 · 30s
  • Webhooks (proven). "Signed Hyperswitch events reach an HMAC-verifying ingress, which forwards the untouched body to the Cloud Run receiver. Cloud Run verifies the HMAC again and settles the payment in Cloud SQL, deduped by event id." I confirmed it in the ingress and Cloud Run logs and the events table, for real payments. The low-level reason a direct run.app delivery does not land is still unconfirmed.
  • Off‑session MIT (the one honest edge). It is coded, but connector‑blocked: Stripe returns UE_9000 until the account gets raw‑card API access. Fauxpay returns no reusable method.
  • Deferred. The automatic renewal scheduler, where the path is already tested, and AP2 cryptographic mandate signatures.
7Close7:00 · 15s
  • "Agents are becoming the customer. Metanoia is the settlement layer for when an agent actually buys, on Hyperswitch."
  • "And the one guarantee under all of it: the number that gets charged is computed by deterministic, unit‑tested server code, on data the model never authored."
